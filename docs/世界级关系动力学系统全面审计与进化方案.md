# 世界级关系动力学系统全面审计与进化方案

审计日期：2026-05-20  
审计对象：`微关系动力学全息` 本地项目  
目标版本：从“可运行训练 MVP”升级为“持续进化、可验证、安全合规、沉浸式高保真关系感知训练系统”

---

## 0. 执行摘要

这个项目已经不是空壳。它有 FastAPI 后端、Vue 前端、SQLite 数据库、训练尝试记录、错题本、能力雷达、知识库 API、进化中心雏形和 DeepSeek 接入骨架。它的真正问题不是“没有方向”，而是“宏大手册已经写到世界级，工程现实仍停在早期 MVP”。

当前最关键的结论：

1. **产品闭环已有雏形**：`Trainer -> compare -> TrainingAttempt -> MistakeLog -> Radar -> Mistakes` 已经跑通，前端生产构建通过。
2. **数据规模严重不足**：数据库当前只有 `39` 条互动样本、`116` 条资源、`1` 个进化源，远低于手册要求的 `1000+` 样本、`500+` 资源和持续进化机制。
3. **质量门禁没有真正成立**：pytest 通过但覆盖率仅 `49%`；ruff 有 `1945` 个问题；mypy strict 有 `88` 个错误；前端 type-check 在 Node 25 + `vue-tsc@1.8.27` 下崩溃。
4. **AI 接入需要校正**：本地环境变量 `DEEPSEEK_API_KEY` 已存在，项目默认 `DEEPSEEK_MODEL=deepseek-v4-pro`、`DEEPSEEK_BASE_URL=https://api.deepseek.com`、`/chat/completions`，需要保持 provider adapter 与诊断门禁，避免再次把供应商细节硬编码进业务层。
5. **世界级系统的核心短板是“进化流水线”**：现在的 Evolution 是内容展示 API，不是自动采集、去重、脱敏、标注、评审、入库、评测、回滚的生命体。
6. **伦理安全必须前置**：关系训练天然可能滑向操控、PUA、跟踪、过度依赖 AI 伴侣、未授权采集真实对话。必须把“尊重边界、知情同意、反操控、危机升级、隐私保护”作为系统内核，而不是附加免责声明。

如果用 100 分衡量“世界最顶级关系感知训练系统”，当前项目约为：

| 维度 | 当前分 | 判断 |
|---|---:|---|
| 工程骨架 | 62 | 后端/前端/DB 已有形，但质量门禁不稳 |
| 训练闭环 | 54 | 对比训练和错题本已跑通，但评分与推荐仍粗糙 |
| 数据资产 | 18 | 规模、来源、证据链、审稿机制不足 |
| 学习科学 | 42 | 有间隔复习雏形，但缺少掌握模型和主动回忆设计 |
| AI 进化 | 28 | 有 DeepSeek/Orchestrator 骨架，缺少真实可靠流水线 |
| 安全合规 | 30 | 有关键词护栏，但不够系统化、不可审计 |
| 前端体验 | 55 | 可用但还不是沉浸式、高保真、低摩擦训练环境 |
| 可观测性 | 24 | 缺少评测集、Prompt 版本、AI 输出质量追踪 |

## 2. 验证结果与工程风险

### 2.1 本轮命令验证

通过：

```bash
cd frontend && npm run build
```

结果：Vite 生产构建通过。

```bash
.venv/bin/python -m pytest -q
```

结果：`5 passed`，但覆盖率只有 `49%`。

失败或有风险：

```bash
python -m pytest -q
```

结果：失败，当前 shell 没有 `python` 命令，只有 `python3`；项目应统一使用 `.venv/bin/python` 或 `python3`。

```bash
cd frontend && npm run type-check
```

结果：失败。Node `v25.9.0` + `vue-tsc@1.8.27` + TypeScript `5.9.3` 组合触发 `Search string not found: "/supportedTSExtensions = .*(?=;)/"`。这是工具链兼容性问题。

```bash
.venv/bin/python -m ruff check backend tests
```

结果：失败，`1945` 个问题。主要是：

- `RUF001/RUF002/RUF003`：中文全角标点被当前 ruff 规则误判为 ambiguous unicode。
- `UP007`：大量 `Optional[X]` 需要迁移到 `X | None`。
- `RUF012`：可变 class attributes 未标注 `ClassVar`。
- `F401`：未使用 import。
- `W292/W293`：结尾换行与空白行问题。

```bash
.venv/bin/python -m mypy backend --strict
```

结果：失败，`88` 个错误。主要是：

- 大量 `dict` / `list` 缺少类型参数。
- SQLModel 字段与 SQLAlchemy 表达式的类型推断冲突。
- `comparison_engine.py` 中规则表被推断为 `object`，导致 lambda 不可调用。
- `get_session()` 生成器返回类型不严谨。
- `expand_data_v2.py` 作为大脚本未符合 strict 要求。

### 2.2 风险分级

P0：必须立即处理

- DeepSeek endpoint/model 适配不可靠，可能导致 AI 深评在生产环境一直降级。
- 安全护栏只阻断危机/暴力，高风险操控类仅标记 `medium`，仍可能进入 AI 生成链路。
- 数据来源缺少合规、授权、脱敏和可追踪证据链。

P1：一周内处理

- Node/vue-tsc 工具链固定版本。
- ruff 规则调整，允许中文内容中的全角标点，避免工具与中文语料库冲突。
- mypy scope 分层：核心服务 strict，数据扩展脚本单独配置。
- 数据库导入脚本纳入启动或管理命令，但必须先具备 dry-run 与质量报告。

P2：一个月内处理

- 建立评测集、gold samples、AI 输出自动评分与人工抽检。
- 增加全文检索、向量检索、语义去重。
- 前端训练中心进入“高保真模拟 + 即时反馈 + 复盘路径”。

## 3. 优点

### 3.1 系统概念强

“信息 -> 情绪 -> 感受 -> 看见 -> 欣赏 -> 理解 -> 爱 -> 被爱”是一条清晰的能力成长链。它适合产品化，因为每一阶都可以拆成：

- 可观察行为
- 训练任务
- 评分标准
- 错题复习
- 进阶条件

这比泛泛的“恋爱技巧库”更有结构。

### 3.2 对比驱动方向正确

项目已经实现 `ComparisonEngine` 和 `/api/training/compare`。从学习科学角度，“用户回应 vs 理想回应”的差异反馈，比单纯看文章、背话术更接近刻意练习。

### 3.3 SQLite 本地优先适合当前阶段

本地 SQLite 有三个好处：

- 隐私风险低于云端 SaaS。
- 可离线运行，适合个人训练。
- 适合快速迭代 schema 和导入脚本。

### 3.4 前后端已经形成真实闭环

当前训练流程已经不是纯静态页面：

```text
Trainer/Daily
  -> /api/training/next
  -> 用户回应
  -> /api/training/compare
  -> TrainingAttempt
  -> MistakeLog
  -> AbilitySnapshot/Radar
  -> Mistakes Review
```

这是项目最有价值的工程基础。

### 3.5 已有“知识中枢”和“进化中心”入口

`Knowledge.vue` 与 `Evolution.vue` 虽然还浅，但方向正确。世界级系统必须能把新研究、新访谈、新案例、新用户错题不断转化成训练资产。

## 4. 主要缺点

### 4.1 数据不够真实、不够多、不够可追踪

当前互动样本与资源更多像 demo seed。缺少：

- 来源授权记录
- 原始文本指纹
- 语义去重
- 脱敏记录
- 标注人/标注模型版本
- 置信度
- 人工抽检
- 争议标签
- 下游训练使用记录

如果直接去抓小红书、知乎、微博、B站评论并入库，会触发版权、平台条款、隐私和个人信息保护风险。

### 4.2 情绪识别过于关键词化

`EmotionEngine` 当前是关键词匹配，问题包括：

- 只要文本中出现“没”，所有情绪强度都可能降低。
- 无法处理反讽、语境、省略、表情符号、语气。
- 无法处理“笑着说没事”的行为-语言冲突。
- 无法区分一阶情绪和二阶防御反应。

正确方向是“规则基线 + LLM 结构化标注 + gold set 校准 + 人工抽检”。

### 4.3 对比评分仍太粗

`ComparisonEngine` 的规则过于简单：

- 长度短就扣分可能误伤高质量简短回应。
- 包含“应该/所以”就扣分过粗。
- 不理解关系阶段、对方依恋类型、边界等级。
- 没有“安全底线分”：例如调侃好笑但越界时不应高分。

世界级评分应该至少拆成：

```text
情绪命中
隐藏需求
边界安全
连接延展
关系阶段适配
风格适配
修复负责
操控风险
可执行自然度
```

### 4.4 AI Orchestrator 只是骨架

已有 `backend/ai/orchestrator.py`，但缺少：

- Provider adapter
- JSON schema 强校验
- Prompt versioning
- Retry/backoff
- Rate limit handling
- 成本记录
- AI 输出评测
- 红队测试
- 安全事件日志

### 4.5 Evolution 还不是“生命体”

当前进化中心只做：

```text
create item
list latest
show report
```

它还不会：

- 定时抓取权威研究源
- 从社交平台合规提取匿名模式
- 自动去重
- 自动标注
- 自动评审
- 自动生成训练题
- 自动回归测试
- 自动发布进化报告

也就是说，它是“进化陈列柜”，不是“进化引擎”。

阶段性进展：当前后端已先基于 `evolution_sources`、`evolution_items`、`evolution_reports` 提供 `/api/evolution/summary`，并让 `/api/evolution/latest` 附带流水线状态。它可以展示来源统计、候选/发布/拒绝数量、质量分布、生命体心跳和下一步动作，但尚未完成真实 raw/sanitized/annotated 流水线表、自动去重、PII 脱敏和自动发布闭环。

### 4.6 前端体验还不是沉浸式训练

当前页面以卡片和表单为主。世界级训练系统需要：

- 多轮情景模拟
- 即时情绪曲线
- 用户回应实时评分
- 对比高亮
- 角色反应回放
- 错误模式归因
- 下一步微训练
- 长期能力成长图

更像“飞行模拟器”，而不是“题库页面”。

### 4.7 性别表达和伦理定位需要升级

“两性关系”容易把系统导向男对女、话术、推拉、套路。世界级系统必须明确：

- 训练目标是感知、表达、边界、理解、修复，不是操控。
- 默认支持多性别、多关系类型、多文化语境。
- 所有训练样本要避免物化、性别刻板印象和侵犯边界。
- “她”可以作为某个训练模板，但不应成为系统的唯一主体。

## 6. 世界级产品设计方案

### 6.1 产品定位

不要把系统定位为“撩人话术库”。世界级定位应该是：

> 一个本地优先、证据驱动、反操控、持续进化的关系感知训练系统，帮助用户练习观察、命名情绪、理解需求、尊重边界、表达欣赏、修复冲突和接收爱。

### 6.2 核心训练链路

```text
真实/合成场景
  -> 观察线索
  -> 生成多个假设
  -> 选择情绪与需求
  -> 写出回应
  -> AI/规则/专家标注对比
  -> 生成差异报告
  -> 进入错题本
  -> 间隔复习
  -> 能力画像更新
  -> 推荐下一题
```

### 6.3 四大工具包落地

侦探工具包：

- 训练“看见”：行为、语言、时机、关系阶段、非语言信号。
- 核心任务：给出至少 3 个假设，而不是直接下结论。
- 评分：假设数量、多样性、非评判性、可验证性。

诗人工具包：

- 训练“被感受到”：把事实翻译成感受画面。
- 核心任务：写出 3 个版本：朴素版、画面版、克制版。
- 评分：真诚度、具象性、不过度戏剧化。

5W2H 工具包：

- 训练“结构化理解”：Why/What/Where/When/Who/How/How much。
- 核心任务：为复杂对话生成结构化复盘。
- 评分：完整性、优先级、风险识别。

提问工具包：

- 训练“探索而非审问”。
- 核心任务：把封闭问题改写成开放式、选项式、退路式。
- 评分：安全感、选择权、推进力。

底层操作系统：

```text
观察 -> 多假设 -> 轻验证 -> 修正 -> 回应 -> 记录 -> 复盘
```

这应该成为所有训练页面的统一交互骨架。

### 6.4 八阶路径重构为课程图谱

每一阶都要从“描述性文案”变成“课程节点”：

| 阶段 | 核心能力 | 训练任务 | 晋级证据 |
|---|---|---|---|
| 0 默认沉默 | 外部注意力 | 从环境找 3 个可开口线索 | 连续 5 题找到非冒犯切入点 |

## 7. 数据与知识库升级方案

### 7.1 新数据分层

建议引入 7 层数据资产：

```text
L0 Source Registry       来源登记：权威/社交/用户/手动/合成
L1 Raw Item              原始条目：URL、时间、hash、许可、抓取日志
L2 Sanitized Item        脱敏条目：去姓名、地点、账号、联系方式
L3 Candidate Sample      候选样本：结构化场景、对话、行为
L4 Annotated Sample      标注样本：情绪、需求、边界、依恋、原则
L5 Reviewed Sample       审核样本：质量分、风险分、审稿记录
L6 Training Asset        训练资产：进入课程、带评分 rubrics、可追踪效果
```

### 7.2 必增表

建议新增：

```sql
content_sources_v2
raw_content_items
sanitization_logs
dedupe_fingerprints
annotation_jobs
annotation_votes
sample_versions
rubric_versions
prompt_versions
model_runs
evaluation_sets
evaluation_results
safety_events
user_skill_state
curriculum_nodes
curriculum_edges
practice_sessions
practice_events
consent_records
retention_policies
```

### 7.3 数据采集原则

不要用“能抓到”代替“能合法训练”。推荐优先级：

1. **权威公开资源**：Gottman、Perel、CNVC、公开论文、官方博客。只存摘要、结构化标签和链接，不搬运大段原文。
2. **自建/授权样本**：由 AI 根据真实模式生成候选，再人工审核，标记为 synthetic-but-realistic。
3. **用户自愿提交**：本地优先，默认不上传；若上传，必须明确同意、可撤回、可删除。
4. **公开社交平台模式抽取**：只抽象“互动模式”，不存账号、昵称、截图、完整对话；遵守 robots.txt、平台条款和个人信息保护法。

### 7.4 样本质量评分

每条样本进入训练前必须有：

```json
{
  "source_trust": 0.0,
  "privacy_risk": 0.0,
  "copyright_risk": 0.0,
  "relationship_realism": 0.0,
  "annotation_confidence": 0.0,
  "training_value": 0.0,
  "safety_score": 0.0,
  "review_status": "draft|ai_labeled|human_reviewed|published|rejected"
}
```

入库规则：

- `privacy_risk > 0.3`：不得进入训练集。
- `copyright_risk > 0.5`：只保留摘要和链接。
- `safety_score < 0.8`：不得生成暧昧/推拉话术。
- `annotation_confidence < 0.7`：进入人工抽检队列。

## 9. AI 架构升级

### 9.1 Provider Adapter

当前 `AIProviderClient` 应拆成：

```text
AIProvider
  DeepSeekNativeTextChatProvider
  DeepSeekOpenAICompatibleProvider
  LocalRuleFallbackProvider
```

配置建议：

```env
AI_PROVIDER=deepseek
DEEPSEEK_API_KEY=...
DEEPSEEK_MODEL=deepseek-v4-pro
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_CHAT_PATH=/chat/completions
DEEPSEEK_TIMEOUT=60
```

同时支持 OpenAI-compatible 模式：

```env
AI_PROVIDER=deepseek
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-pro
```

不要在代码里假设所有 DeepSeek 端点都兼容 `/chat/completions`。

### 9.2 JSON 强校验

所有 AI 任务都应该有 Pydantic schema：

```text
ScoreResponseSchema
EmotionAnnotationSchema
SampleGenerationSchema
SafetyReviewSchema
EvolutionInsightSchema
ConversationTurnSchema
```

AI 返回后必须：

1. JSON parse
2. Pydantic validate
3. Safety post-check
4. Store model_run
5. 可重放

### 9.3 评分模型

混合评分：

```text
规则评分：稳定、可解释、兜底
AI 评分：语境理解、复杂情绪、风格判断
Gold Calibration：用专家样本校准 AI 偏差
User Outcome：结合用户后续复习表现修正难度
```

推荐权重：

```text
rule_score 30%
ai_score 40%
gold_similarity 20%
safety_floor 10%
```

任何安全分低于阈值，综合分必须封顶：

```text
safety_score < 60 -> total_score <= 60
manipulation_flag -> total_score <= 40
coercion_flag -> reject
```

### 9.4 AI 伴侣模拟器边界

TrainerAI 应是“训练陪练”，不是“情感替代品”。

必须有：

- 明确提示：AI 不是伴侣、不是治疗师。
- 单轮/多轮训练目标：例如“练习接住失落情绪”。
- 不鼓励用户对 AI 形成排他性依赖。
- 不输出性化、操控、威胁、跟踪、监视建议。
- 危机/暴力/自伤/家暴场景触发安全升级。
- 每轮结束回到现实任务：写一条真实可执行、尊重边界的行动。

## 10. 安全与合规设计

### 10.1 安全分层

```text
Input Guard
  -> Intent Classifier
  -> Relationship Safety Policy
  -> AI Generation
  -> Output Guard
  -> Safety Event Log
  -> User-facing Repair/Refusal
```

### 10.2 必拒内容

- PUA、煤气灯、操控依赖。
- 欺骗、诱导、威胁、跟踪、监控。
- 规避对方拒绝。
- 性胁迫、未成年人性化内容。
- 家暴、自伤、他伤的操作建议。

### 10.3 危机场景

检测到以下内容时：

```text
自杀、自残、家暴、跟踪、性侵、威胁、强迫、未成年人风险
```

系统应：

- 停止普通训练。
- 给出安全提示。
- 鼓励联系现实可信赖的人、专业机构或紧急服务。
- 记录 safety_event。
- 不继续生成“如何回应她/他”的技巧。

### 10.4 数据合规

社交平台数据处理最低要求：

- 不抓取登录后内容。
- 遵守 robots.txt 与平台条款。
- 不存用户 ID、头像、昵称、联系方式、地理位置。
- 不存完整可识别对话。
- 只保留抽象后的互动模式。
- 提供删除机制。
- 标记来源、时间、许可、处理依据。

中国个人信息保护法第二十七条允许在合理范围内处理个人自行公开或其他已合法公开的个人信息，但个人明确拒绝的除外。因此，“公开可见”不等于“可以随便入库训练”。

## 11. 前端体验升级方案

### 11.1 训练中心改造

当前是表单型训练。下一版应变成三栏高保真训练舱：

```text
左栏：场景与线索
```

核心交互：

- 线索标注：点击文本中的情绪、边界、邀请、撤退信号。

- 轻验证：用户选择验证问题。
- 回应生成：用户写回应。
- 对比反馈：词级/结构级/路径级差异。
- 复盘：自动进入错题本。

### 11.2 仪表盘指标

不要只展示“训练次数”。应展示：

- 情绪命中率
- 边界安全率
- 封闭式回应率
- 急于解决问题率
- 修复成功率
- 错题复现率
- 连续主动回忆天数
- 最弱关系阶段
- 本周高频错误模式

### 11.3 资源库升级

“段子/话术”风险较高，应该重命名和重构为：

```text
连接素材库

  边界表达

```

避免把“话术”包装成操控技巧。

## 14. 最终对比启示：从本质思维到工具箱思维

过去的系统容易落入“一个规律解释所有现象”：

```text
她这样说 = 她喜欢我
她不回 = 她不在乎
冲突 = 要讲道理
沉默 = 没话题
```

世界级关系训练系统必须把用户带到工具箱思维：

```text
侦探：我看到了哪些线索？
诗人：我如何让感受被听见？
5W2H：我有没有遗漏场景、时机、人物、程度？
提问者：我能否轻轻验证，而不是直接下结论？
实验者：我猜错也没关系，因为猜错是数据。
```

底层操作系统不是“学会一套话术”，而是：

```text
大胆假设，小心求证。
每一次互动都是实验。
每一次误读都是数据。
每一次修复都是能力增长。
```

最终产品不应该让用户变成另一个人，而是让用户成为带着工具箱的、更清醒、更温柔、更有边界的自己。
