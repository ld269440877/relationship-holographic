# 功能模块选项卡结构模板规范

## 目标

所有功能模块的选项卡都必须帮助用户回答三个问题：

1. 我现在在哪个模块？
2. 这个选项卡解决什么具体任务？
3. 我点进去以后能看到什么、做什么、如何判断完成？

选项卡不是装饰，也不是把页面切碎。它必须把复杂模块拆成清晰、可扫描、可执行、可退出的工作区。

## 通用结构

每个模块页面建议使用同一套结构：

```text
模块标题
-> 模块一句话目标
-> 关键指标 2-4 个
-> 选项卡导航
-> 当前 tab 的任务说明
-> 当前 tab 的主体内容
-> 右侧目录或上下文帮助
-> 空状态 / 加载 / 错误 / 权限不足
```

每个 tab 必须具备：

| 字段 | 用途 | 示例 |
|---|---|---|
| `id` | 稳定标识，进入 URL query 或本地状态 | `cases` |
| `label` | 用户看得懂的短名称 | `案例训练` |
| `summary` | 解释这个 tab 解决什么问题 | `看完整案例并做迁移练习` |
| `primaryContent` | 主体内容类型 | `资源卡列表 / 详情 / 图谱 / 表格 / 对话流` |
| `primaryAction` | 用户最自然的下一步 | `加入训练` |
| `emptyState` | 没数据时说什么 | `当前筛选暂无案例，试试清空标签` |
| `qualityGate` | 这个 tab 达标的底线 | `必须显示场景、原话、错误回应、更好回应` |

## 视觉和交互规则

- 选项卡使用横向 segmented control，移动端可横向滚动。
- tab 文案控制在 2-6 个汉字，解释放在摘要里。
- 当前 tab 必须有明显高亮、`aria-selected` 和键盘可访问状态。
- tab 下方第一屏必须显示“当前 tab 做什么”，不能只出现一堆卡片。
- 多记录 tab 必须接右侧可折叠目录或列表内锚点。
- 每个 tab 都要有空状态、加载状态、错误状态。
- 不在 tab 内再嵌套另一个大 tab；需要二级筛选时用筛选器、chip 或分组标题。

## 模块模板

### 资源海洋 `/resources`

| Tab | 目的 | 必备内容 | 主动作 |
|---|---|---|---|
| 全部资源 | 综合浏览所有 published/reviewed 资源 | 类型、场景、质量分、来源边界 | 查看详情 |
| 案例训练 | 只看完整案例卡 | 场景、TA 原话、常见失误、更好回应、练习任务 | 加入训练 |
| 表达工具 | 按表达工具反查资源 | 工具名、工具链、适用场景 | 打开工具 |
| 来源与许可 | 看来源、链接健康和使用边界 | 来源标题、链接、摘要、许可、健康状态 | 打开来源 |
| 治理队列 | 看重复、低质、隔离资源 | 重复簇、质量分、状态、批次 | 提交复核 |

## 空状态模板

```text
当前没有可展示的{对象}。
可能原因：筛选太窄、数据还未发布、该来源需要复核。
你可以：清空筛选 / 查看全部 / 进入治理队列 / 创建练习。
```

## 验收清单

1. 每个模块至少有 3 个稳定 tab，且 tab 名称不重复、不抽象。
2. 每个 tab 有明确任务、主体内容、主动作和空状态。
3. 搜索、筛选、目录、详情、训练入口不互相抢位置。
4. tab 状态可进入 URL query 或稳定本地状态，刷新后不丢失。
5. 移动端不换行挤压，横向滚动但当前 tab 可见。
