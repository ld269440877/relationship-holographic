## 1. 项目摘要

“微关系动力学全息”是一个面向亲密关系沟通训练、关系信号识别和安全回应练习的关系感知训练系统。目标用户包括希望提升关系沟通能力的个人用户、情感教育/咨询从业者、训练内容运营者和系统维护者。核心价值主张是把关系互动从模糊经验转化为可训练、可复盘、可审计、可持续进化的数据闭环：用户端获得沉浸式练习与 AI 伴侣反馈，管理端获得内容治理、Gold Set 校准、导入质量和运营审计，开发者端获得 SQLite 主真源、API、迁移、调度、向量检索和自动化质量门禁。当前核心交付物包括 Web 前端、FastAPI 后端、SQLite 数据底座、AI 安全编排、训练/复盘闭环、进化流水线、分析中心、审计中心、发布治理台和自动化 smoke/pytest 质量门禁。

## 5. 缺陷分析与优化改进方案

### 5.1 AI Provider 生产稳定性不足

- 缺陷描述：DeepSeek 历史日志存在 HTTP 400 高频，当前成功率偏低。系统已判断本地请求形态无明显风险，但账号区域、模型授权、服务侧参数约束仍未通过真实授权探针闭环。
- 影响等级：高。
- 优化步骤：
  1. 负责人建议：后端负责人 + 客户技术联系人。
  2. 客户提供受控测试环境 API key、账号区域、模型授权截图和允许 live probe 的时间窗。
  3. 设置 `DEEPSEEK_LIVE_PROBE_ENABLED=true`，先执行 dry-run，再执行最小 live probe。
  4. 将 HTTP status、error type、latency、request shape 写入 `ai_provider_probe_logs`。
  5. 若仍 400，按 failure playbook 分类为账号授权、模型不可用、区域限制或参数约束。
  6. 固化成功/失败样例到 `tests/test_ai_provider_safety.py`。
  7. 回归 `tools.commander regression-audit`。
- 工作量预估：1～2 人日，不含客户开通/授权等待。
- 验收标准：最近 30 次受控探针成功率 ≥ 90%；失败日志无 prompt/API key/响应全文泄露；fallback 可用。
- 预期效果：明确 Provider 是否可进入生产能力，避免长期依赖模糊 fallback。

### 5.2 Gold Set 专家覆盖与真实专家身份仍需业务闭环

- 缺陷描述：系统已有复核、冲突治理和一致率指标，但“专家复核”真实性、样本规模、专家间一致性规则仍需客户制度确认。
- 影响等级：高。
- 优化步骤：
  1. 负责人建议：内容负责人 + 领域专家 + 产品负责人。
  2. 确认专家资质、审阅者 ID 规则、最小专家数、样本抽样策略。
  3. 将 Gold Set 分层：基础沟通、边界、冲突、修复、亲密推进、高风险安全。
  4. 每层至少抽取 30～50 条样本进入专家复核。
  5. 使用现有 `gold/summary`、`gold/interrater`、`conflict-queue` 查看覆盖率和一致率。
  6. 对冲突样本生成 consensus_review，但保留原审阅历史。
  7. 设定发布门禁：覆盖率、平均信心、一致率、分差同时达标才可纳入严格训练。
- 工作量预估：3～8 人日，取决于专家可用性。
- 验收标准：专家覆盖率 ≥ 70%；决策一致率 ≥ 0.8；平均总分差 ≤ 8；开放冲突为 0。
- 预期效果：提升训练评分可信度，减少伪专家/伪金标风险。

### 5.3 导入历史 issue 仍有人工来源治理债务

- 缺陷描述：自动可修字段已清零，但历史来源 issue 仍需人工确认，不能通过代码刷分。
- 影响等级：中。
- 优化步骤：
  1. 负责人建议：数据运营负责人。
  2. 在 `/governance` 按 source group 查看来源复核包。
  3. 对 warning-only 来源执行批量 dry-run，确认 impact、source_counts、severity_counts。
  4. 人工填写 reviewer 与 resolution。
  5. apply 后查看 issue audit 历史，确认 resolution hash 与日志 ID。
  6. 每批执行后跑 `regression-audit --batch-limit 2`。
  7. 对 error 来源先 request_review，不直接 resolve。
- 工作量预估：2～5 人日。
- 验收标准：active issue 降至 ≤ 10；resolved issue 有审计日志；无 resolution 明文泄漏。
- 预期效果：导入质量分从 75 左右向 85+ 提升，来源治理可追溯。

### 5.4 治理台操作交互 smoke 尚未覆盖 dry-run/apply

- 缺陷描述：已覆盖训练失败路径，但 `/governance` 的发布操作、Issue dry-run/apply 还未纳入交互 smoke。
- 影响等级：中。
- 优化步骤：
  1. 负责人建议：前端负责人。
  2. 在 `world-class-smoke.mjs` 增加 `reviewed-assets/action` mock。
  3. 增加 `import-quality/issues/action` mock。
  4. 桌面/移动分别选择候选、点击 Dry-run、验证预演结果。
  5. 点击 apply，验证审计日志 ID、状态变化和错误为空。
  6. 对 Issue 操作重复上述流程。
  7. 将检查项纳入 report。
- 工作量预估：0.5～1 人日。
- 验收标准：smoke checked 项增加；治理台 dry-run/apply 在桌面/移动均通过；无横向溢出/重叠/runtime error。
- 预期效果：治理台从“页面可用”提升到“关键操作可回归”。

### 5.5 工作区变更边界与发布管理需整理

- 缺陷描述：工作区存在大量未跟踪/修改项，长期不拆分会影响审计、回滚和交付。
- 影响等级：高。
- 优化步骤：
  1. 负责人建议：技术负责人。
  2. 创建 `codex/stabilize-functional-reliability` 分支。
  3. 按后端数据、前端体验、测试门禁、文档状态拆分提交。
  4. 将生成物如 `.coverage`、`frontend/dist`、`frontend/output` 判断是否纳入 `.gitignore` 或作为审计资产保留。
  5. 对未跟踪中文素材确认是否都已归档到 `content_sources/`。
  6. 建立 release checklist：pytest、ruff、mypy、type-check、build、smoke、commander sync。
- 工作量预估：1～2 人日。
- 验收标准：git status 可解释；提交粒度清楚；生成物策略明确；回滚路径可用。
- 预期效果：提高交付可信度，降低多人协作风险。

### 5.6 生产部署与后台调度仍偏本地化

- 缺陷描述：已有 APScheduler 入口、launchd/systemd 文件和 health，但未真正部署为生产守护服务，也未接入外部告警。
- 影响等级：中。
- 优化步骤：
  1. 负责人建议：运维/平台负责人。
  2. 确认目标部署环境：本机、VPS、Docker、NAS、云主机或内网服务器。
  3. 确认 SQLite 备份路径、日志路径、服务用户权限。
  4. 应用 systemd/launchd 配置到系统目录。
  5. 接入失败告警：邮件、Webhook、飞书/Slack/企业微信之一。
  6. 每日自动执行 `scheduler-health` 和 `regression-audit`。
  7. 每周恢复演练一次 SQLite 备份。
- 工作量预估：1～3 人日。
- 验收标准：服务重启后自动恢复；调度任务按计划执行；失败告警可达；备份可恢复。
- 预期效果：从本地开发型系统升级到可持续运营系统。

## 结论

项目已经从原型推进到“可运行、可审计、可持续进化”的较高成熟度：数据底座、训练闭环、AI 安全、Gold Set、内容治理、分析审计、调度和前端可靠性门禁均已有实现。当前最主要的差距不再是普通页面功能，而是三类生产化闭环：真实外部 Provider 授权验证、真实专家/内容来源确认、生产部署与工作区发布管理。工程侧下一步最值得自动推进的是 `/governance` 发布和导入 Issue 操作的交互 smoke，以及生产部署/备份/告警的落地脚本；业务侧必须等待客户对专家、来源、凭证和合规边界作出明确确认。
